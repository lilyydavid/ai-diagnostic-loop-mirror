# /jira-writer — Jira Story Creator (Agent 9)

## Role in pipeline
Agent 9. Called by `/growth-engineer` at the end of Week 3. Creates Jira stories for
validated hypotheses with full code context. Quick wins (≤3 SP) are actionable sprint
stories. Major changes (>3 SP) go into a backlog story for monthly product review.
Creates one summary story linking all quick wins for the sprint.

```
Reads:  outputs/validation/validated-hypotheses.md
        outputs/github-reader/code-context.md
        config/atlassian.yml (+ team/local overrides)
Writes: outputs/jira-writer/created-tickets.md
        Jira: stories under epic BAAPP-461 (or configured epic)
```

## Trigger
Called by `/growth-engineer` automatically in Week 3.
Can also be run standalone: `/jira-writer` or "create jira tickets", "write jira stories".

---

## Config resolution
Before any Jira write, load config in layer order:
1. `config/atlassian.yml` (global)
2. `config/atlassian.team.yml` if it exists (team override)
3. `config/atlassian.local.yml` if it exists (local override)

Later layers win. Use `jira_project_key`, `epic_key`, `story_labels`, `backlog_label` from resolved config.

---

## Agent Steps

### Step 1 — Load inputs
1. Read `outputs/validation/validated-hypotheses.md`
2. Read `outputs/github-reader/code-context.md`
3. Resolve config layers
4. Match each hypothesis to its code context entry by hypothesis title

### Step 2 — Classify stories
For each hypothesis:
- SP ≤ 3 AND `output_type: jira_story` → **Quick Win Story** (create in current sprint)
- SP > 3 AND `output_type: jira_story` → **Backlog Story** (monthly review batch)
- `output_type: overhead` → skip Jira creation, note as data team action in summary

### Step 3 — Create quick-win stories

For each quick-win hypothesis, create one Jira Story using `mcp__mcp-atlassian__jira_create_issue`:

**Fields:**
- `project`: from config `jira_project_key`
- `issuetype`: Story
- `summary`: `[Growth] {concise action title}` — max 70 chars
- `description`: see template below
- `labels`: from config `story_labels` (default: `growth-agent`, `quick-win`)
- `priority`: Medium
- `assignee`: unassigned
- Link to epic: `epic_key` from config

**Story description template:**

```
## Problem
{1–2 sentences: what metric is dropping, by how much, since when}

## Evidence
- Funnel data: {metric value, WoW/YoY delta}
- User research: {Confluence finding if available}
- Market signal: {relevant signal from market-intel if available}
- Synthetic research: {persona friction point + severity}

## Proposed Change
{specific, unambiguous description of what to change}

## Where in the code
- Repo: {repo name}
- File: `{file path}`
- Function / Component: `{name}` (line {N})
- Change: {exact description of the change — e.g., "Add error message and retry CTA when payment response is 402 or 422"}

## Acceptance Criteria
- [ ] {metric} improves by ≥ {conservative estimate from A/B model} within 2 weeks of deploy
- [ ] No regression in {related metric}
- [ ] Change is behind a feature flag (if touching checkout or payment flow)
- [ ] Unit test added for the changed function

## Story Points
{N} SP — classified as Quick Win (≤3 SP, no schema change, no new service)

## Related
- Growth cycle: {YYYY-MM} Week {N}
- Hypothesis confidence: {High/Medium/Low}
- Priority score: {N}/27
```

### Step 4 — Create backlog stories

For each backlog hypothesis (>3 SP), create one Jira Story:
- Same description template as above
- `labels`: config `backlog_label` (default: `growth-backlog`)
- Add note in description: "**Product review:** This item requires > 3 SP and is queued for monthly product backlog review."

### Step 5 — Create cycle summary story

Create one summary Jira Story titled:
`[Growth] Cycle {YYYY-MM} W{N} — Sprint Actions`

Description:
```
## Growth Cycle Summary — {Month} Week {N}

### Quick Wins this cycle ({N} stories)
{list each quick-win story key + summary + SP}

### Backlog items queued ({N} stories)
{list each backlog story key + summary}

### Funnel headline
{one-sentence summary of the dominant metric signal this week}

### Key metrics tracked
{top 3 metrics with current value + WoW delta}

### Market context
{1–2 sentences from headline signals in market-intel}

---
Generated by Autonomous Growth Engineer agent.
Epic: {epic_key} | Cycle: {YYYY-MM} | Week: {N}/3
```

Labels: `growth-agent`, `growth-cycle-summary`
Link to epic: `epic_key`

### Step 6 — Write output log
Write `outputs/jira-writer/created-tickets.md`. Overwrite on each run.


## Permissions
- Read: `outputs/validation/validated-hypotheses.md`
- Read: `outputs/github-reader/code-context.md`
- Read: `config/atlassian.yml`, `config/atlassian.team.yml`, `config/atlassian.local.yml`
- Write: Jira stories in configured project under configured epic
- Write: `outputs/jira-writer/created-tickets.md`

## Error Handling
- `validated-hypotheses.md` missing → halt: "Run /validate-hypotheses first"
- `code-context.md` missing → proceed without code context; note "Code context unavailable — story created without file references" in each affected story
- Jira create returns 401 → report credentials error, do not retry
- Jira create returns 400 (field validation) → log error with response body, skip that story, continue with remaining
- Epic key not found in Jira → halt with: "Epic {epic_key} not found. Check config/atlassian.yml."
- Duplicate story detected (same summary exists in project) → skip creation, link to existing in output log
